Between 1985 and 1987, a radiation therapy machine called the Therac-25 killed three people and seriously injured three others. It was supposed to deliver controlled doses of radiation to cancer patients. Instead it delivered doses hundreds of times higher than intended.

The cause was a race condition in the C code controlling the machine — exactly the kind of concurrency error that CERT C rules are specifically designed to prevent. This is what happens when C code controlling life-critical systems has bugs. People die. And the industries that learned this the hard way are now desperate for developers who know how to write secure C.

Aerospace: Where Every Line Gets Audited

Flight control software runs on embedded systems with no room for error. You're at 35,000 feet when the autopilot makes a calculation based on sensor data stored in memory. If a buffer overflow corrupts the memory location next to that sensor data, the autopilot is now making decisions based on garbage values.

The gets() function copies characters into a buffer regardless of size. If you're reading sensor data and fail to validate array bounds, you can overwrite adjacent memory storing your altitude. Or your airspeed. Or your pitch angle.

When aerospace engineers read CERT C Rule ARR30-C — "Do not form or use out-of-bounds pointers or array subscripts" — they're not reading a style guideline. They're reading a rule written in blood.

Automotive: Your Car is a Computer on Wheels

Modern vehicles have millions of lines of C code controlling anti-lock brakes, airbag deployment, electronic stability control, and autonomous driving features. What happens when a multiplication in the braking calculation overflows?

In C, when you multiply two integers and the result exceeds the maximum value the type can hold, you don't get an error. You get wraparound. A large positive number becomes small — or negative. Then that negative value gets cast to unsigned, and suddenly you have a massive number that makes no physical sense. Your car's computer thinks the vehicle ahead is either impossibly close or impossibly far away.

INTEGER OVERFLOW

The automotive industry requires secure coding practices because integer overflows have caused real crashes with real injuries.

Medical Devices: Code Running Inside Your Body

A pacemaker is a small computer implanted in your chest. The code runs in C because the hardware is resource-constrained and needs direct memory access for real-time performance.

Imagine the code has a null pointer dereference. In a desktop application: segfault, restart. In a pacemaker: the device crashes and stops monitoring heart rhythm. If the patient has a life-threatening arrhythmia in those seconds, there's no one watching. No one delivering the electrical impulse that keeps them alive.

Consider an insulin pump: a simple bounds checking error could cause it to misread glucose values. Too much insulin means hypoglycemic shock, seizure, coma, death. CERT C exists to eliminate undefined behavior before code gets implanted in someone's body.

Industrial Control Systems: When SCADA Fails

SCADA systems were designed in the 1970s and 80s when "security" meant a chain-link fence. They now connect to corporate networks, which connect to the internet. A format string vulnerability in the chlorine monitor logging code can let an attacker write to arbitrary memory addresses and disable the monitors that detect when water isn't properly disinfected.

REAL INCIDENT

In 2021, an attacker accessed a water treatment facility's SCADA system and attempted to poison the water supply by remotely changing chemical levels. The attack was detected and stopped — but it proved these systems are accessible and vulnerable.

IoT: Billions of Targets

39.6 billion devices projected by 2033. Most run embedded C on processors with no memory protection and no runtime bounds checking. Here's firmware for a smart thermostat:

C█
char buffer[64];
gets(buffer);  /* Read the setpoint command */

The same gets() that enabled the Morris worm in 1988. An attacker sends 200 bytes. Your 64-byte buffer overflows. Adjacent memory gets corrupted — maybe the authentication token, maybe function pointers the attacker can redirect to malicious code. Multiply this across millions of IoT devices and you have a massive botnet.

Why These Industries Need You

82% of software vulnerabilities come from coding errors — not zero-days, not sophisticated exploits. Just bugs in the code. The CERT C Coding Standard was developed by over 1,900 security experts who analyzed real-world vulnerabilities and documented the practices that prevent them.

If you're learning CERT C, you're not just learning rules. You're learning the accumulated wisdom from decades of debugging failures that killed people. You're learning how to write code that saves lives.
