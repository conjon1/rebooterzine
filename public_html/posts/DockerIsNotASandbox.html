Docker is Not a Sandbox
Docker is sophisticated plumbing on top of your system — root privileges, a background daemon, namespaces and cgroups — keeping containers contained by convention, not by isolation guarantees. Treating it like a fortress is a common and dangerous mistake.

One of the biggest misconceptions I see about Docker is that it's somehow sandboxed — a place where you can safely test programs and services in isolation. This is simply not true.

Docker is operating-system-level virtualization, not hardware-level virtualization. All containers on a Linux host share the same kernel. They aren't VMs with their own hardware abstraction — they're processes running on the same kernel, isolated by rules, not by hardware. That shared kernel means containers are only as secure as the kernel itself.

Architecture Overview

  1. Docker Daemon (dockerd) — manages images, containers, networks, and volumes. Runs as root by default. Every container operation ultimately touches privileged parts of your system.
  2. Container Runtime (containerd + runc) — actually creates containers, spawning processes in namespaces and applying cgroups.
  3. Images and Filesystems — layered OverlayFS. Each layer is read-only; a writable layer is added when the container runs.

Docker's isolation stops at the kernel boundary. It's a shared playground, not a sealed vault.

Threat Modeling: Component by Component

dockerd — the one process that changes everythingTHREAT

How it's abused:

  • Socket abuse: anything that can write to /var/run/docker.sock can talk to the daemon as admin — create containers, mount host paths, set --privileged, run arbitrary commands. Full host control.
  • Misconfigured access: adding untrusted users to the docker group gives them root-equivalent powers.
  • Remote exposure: exposing the API over TCP opens the same attack surface to the network.
MITIGATIONSSAFE
  • Never mount /var/run/docker.sock into containers. Use a proxy like docker-socket-proxy if needed.
  • Don't add non-admin users to the docker group.
  • Never expose the Docker API over TCP without mutual TLS.
  • Run rootless Docker where feasible.
Namespaces and cgroups — the illusion of separationTHREAT

How it's abused:

  • Kernel exploits: namespaces are a visibility boundary, not a security boundary against kernel bugs. Privilege escalation vulnerabilities can let a container process affect global kernel state.
  • UID 0 mapping: without user namespaces, container root maps to host root. Container root is host root.
MITIGATIONSSAFE
  • Enable user namespaces where feasible so container root is not host root.
  • Avoid --privileged — it removes many namespace protections and grants all capabilities.
  • Use --cap-drop ALL and only add required capabilities with --cap-add.
  • Keep kernels patched aggressively.
Bind mounts — the convenience that bitesTHREAT

How it's abused:

  • Mounting / or /var into a container gives it direct read/write on host data.
  • Device mounts (--device) can allow interaction with hardware that enables kernel attack surfaces.
MITIGATIONSSAFE
  • Avoid host-root mounts unless absolutely necessary. Use managed volumes instead.
  • Mount read-only where possible (:ro).
  • Audit any CI scripts that mount the host for "debugging" — these are common accidental backdoors.

Hardening Checklist

BASH█
docker run --rm \
  --cap-drop ALL \
  --security-opt no-new-privileges:true \
  --security-opt seccomp=/path/to/seccomp.json \
  --read-only \
  -v /data/app:/app:ro \
  your/image:tag
  • Enable seccomp and LSMs (AppArmor/SELinux)
  • Scan images in CI with Trivy or Clair — fail builds on critical findings
  • Pin image digests: your/image@sha256:<digest>
  • Restrict dockerd access — never expose the socket, never add non-admins to the docker group
  • Run rootless Docker for dev and CI

Exploits in the Wild

  • CVE-2019-5736 (runc) — malicious container overwrites the host runtime and gains root
  • CVE-2024-21626 (runc) — file-descriptor leaks allow container to operate on host paths during exec/run
  • CVE-2022-0847 (Dirty Pipe) — kernel LPE usable from inside a container to overwrite host files
  • CVE-2025-31133 (runc 2025) — recent high-severity runc vulnerabilities enabling container escape via mount/race/fd attacks
TAKEAWAY

If you're running code you don't fully control, consider gVisor or Kata Containers instead of bare Docker. The extra isolation layer costs something — but so does a container escape.
