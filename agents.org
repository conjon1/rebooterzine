#+TITLE: System Context & Rules for AI Agents
#+DESCRIPTION: Read this file before making modifications to the Rebooterzine codebase.

* System Overview
You are working on Rebooterzine, a custom static site generator written in Go.
The core philosophy of this project is: **Compile-time safety and speed over runtime parsing.**
Instead of parsing Markdown files, all content is written directly in Go using the `templ` library. The build script (`cmd/build/main.go`) executes these templates and outputs raw HTML files into the `public_html/` directory.

* Directory Structure
- `/cmd/build/main.go` -> The build script. Executes the registry, generates JSONs, and writes to `public_html/`.
- `/model/types.go` -> The central data structures (`Post`, `AuthorData`).
- `/internal/` -> The core Go logic for sorting registries, parsing files, and writing data.
- `/posts/` -> The actual blog content. Each post is a `.templ` file.
- `/components/` -> The UI library. Contains layout shells and reusable `.templ` elements (Code blocks, Terminals, Callouts).
- `/assets/` -> Static files (CSS, JS, images, ASCII art).
- `/public_html/` -> The output directory for all built HTML and JSON files, as well as the manual author profile HTML files.

* Core Patterns & Rules for Agents

** 1. The Post Registry Pattern
DO NOT modify `registry.go` when creating new posts.
Posts are self-registering. When asked to create a new post, you MUST use the `init()` pattern.
#+BEGIN_SRC go
func init() {
	Register(model.Post{
		ID:       "unique-slug",
		Title:    "Post Title",
		Date:     "Month DD, YYYY",
		Author:   "Author Name",
		AuthorID: "authorid",
		Excerpt:  "Summary.",
	}, FunctionName)
}
#+END_SRC

** 2. Templ Lexer Quirks (CRITICAL)
The `templ` parser has known bugs when mixing Go templates with large blocks of English prose. You MUST obey these rules when writing inside a `.templ` file:

- **Rule 2A (Apostrophes):** NEVER use raw apostrophes/single quotes (`'`) in text paragraphs. The parser mistakes them for Go rune literals and fails. You MUST escape them using the HTML entity `&apos;`.
  - *BAD:* `<p>It's my dog's toy.</p>`
  - *GOOD:* `<p>It&apos;s my dog&apos;s toy.</p>`
- **Rule 2B (String Literals):** When passing strings to components (like `@components.Callout("info", "...")`), prefer double quotes (`" "`) over backticks (`` ` ` ``) unless the string contains multiple lines of code.
- **Rule 2C (Line Starts):** Do not start a new line of text with a Go keyword (e.g., `if`, `switch`, `for`, `case`). The lexer will try to parse it as logic.

** 3. File Output & Package Purity
- **NEVER** attempt to move generated `_templ.go` files into a separate folder. Go requires all files sharing a `package` declaration to live in the same physical directory.
- `_templ.go` files are ignored via `.gitignore`. Leave them side-by-side with their `.templ` source files.
- The build script (`main.go`) handles outputting the final `.html` files directly into `public_html/posts/`.

* Component Library Exhaustive Reference
Do not write raw inline styles or complex raw HTML for common elements. Utilize the `components` package.

** Layout & Macro Components
These wrap pages and provide the base HTML document structure.
- `layout.templ` -> The root `<html>` shell. Injects global CSS and JS. Wraps all standard pages.
- `post_layout.templ` -> The specific wrapper for blog posts. Injects `post.css`, author metadata headers, and footer navigation blocks.
- `nav.templ` & `footer.templ` -> Global top and bottom navigation bars.
- `ticker.templ` -> Renders a scrolling marquee text bar for alerts or status updates.

** Prose Elements (`elements.templ`)
These are injected directly into post content to create the terminal aesthetic.
- `@components.CodeBlock(lang string, code string)` -> Renders a preformatted code block with syntax highlighting targets and a language label tab.
- `@components.Terminal(prompt string, content string)` -> Renders a fake window with traffic-light window controls (macOS style) and a CLI prompt prefix.
- `@components.Callout(kind string, message string)` -> Renders a bordered alert box. Valid `kind` strings: `"info"`, `"warn"`, `"danger"`, `"tip"`.
- `@components.SectionHeader(title string)` -> Renders an H2 tag prepended with a terminal `$` or `>` prompt to separate major sections.
- `@components.FileTree(tree string)` -> Renders a raw string inside a specialized monospace block designed to look like the output of the UNIX `tree` command.
- `@components.Aside() { ...children }` -> A container for pull-quotes, side tangents, or floating reference material. Pass standard HTML inside the braces.
- `@components.Image(src string, caption string)` -> Renders a responsive `<figure>` and `<img>` tag. Automatically adds a `<figcaption>` if the caption string is not empty.
- `@components.StatusBadge(label string, value string)` -> Renders an inline, high-contrast block badge (e.g., `[ TESTED ON | Ubuntu 24.04 ]`).

* Frontend Architecture: CSS & JS Logic

** CSS Logic (`assets/css/`)
The site uses a brutalist, high-contrast terminal aesthetic. **RULE: Do not use inline `style="..."` attributes.**
- `variables.css` -> The source of truth. Contains all CSS custom properties (colors, monospace font stacks, spacing calculations). Modify this to change the global theme.
- `base.css` -> Global element resets, body background styling, and base typography.
- `post.css` -> Scoped specifically to the prose inside blog posts. It formats standard HTML tags (p, h1, h2, ul, blockquote) to look good in a long-form reading format.
- `prose.css` -> Contains the specific structural styling for the custom `@components` (Terminals, Callouts, Badges).
- `animations.css` -> Contains keyframes for blinking cursors, typing effects, and page transition fades.

** JS Logic (`assets/js/`)
The site uses strict Vanilla JS to keep the payload microscopic. **RULE: Do not introduce React, Vue, jQuery, or other heavy dependencies.**
- `router.js` -> The core SPA-like engine. It intercepts `<a>` tag clicks, prevents default browser navigation, fetches the target HTML file via AJAX, and swaps the `<main>` DOM node. This allows for seamless page transitions without full reloads.
- `search.js` -> Fetches the compiled `posts.json` generated by the backend and provides instant, client-side fuzzy search on the blog index page.
- `typewriter.js` / `boot.js` -> Handles the aesthetic terminal "boot sequence" effects on the homepage, typing out strings character by character.
- `observers.js` -> Uses IntersectionObserver to trigger fade-in animations as elements scroll into the viewport.

* Asset Management: Art, Images, and GIFs

All static visual assets must be placed inside the `assets/` directory (which is typically mirrored or served directly alongside `public_html/`).

** Adding Standard Images & GIFs
1. Place the `.jpg`, `.png`, or `.gif` file into `assets/img/` (or the respective image directory).
2. Inside the `.templ` post file, use the Image component to render it cleanly:
   #+BEGIN_SRC go
   @components.Image("/assets/img/my_animated_demo.gif", "A demonstration of the CLI tool in action.")
   #+END_SRC
   *Note: Using the Image component ensures the GIF scales correctly on mobile and respects the site's border-radius and margin variables.*

** Adding ASCII Art
The site leverages raw `.txt` files for ASCII art to maintain the terminal vibe.
1. Create a new `.txt` file containing your ASCII art (e.g., `assets/ascii/server.txt`).
2. To use it, you can either:
   - Fetch it via JavaScript (`fetch('/assets/ascii/server.txt')`) and inject it into a `<pre>` tag for a typewriter boot effect.
   - Or embed it directly into a component or post using a standard `<pre class="ascii-art">` block.
